# DECISIONS

## D-001 CLI 优先
- 日期：2026-03-25
- 决策：第一版入口使用 CLI，而不是先做 FastAPI
- 原因：更快跑通状态图、工具路由和调试链路，适合阶段 0/1

## D-002 阶段边界固定为阶段 0 + 最小阶段 1
- 日期：2026-03-25
- 决策：当前不跨阶段扩展，不提前做 GraphRAG、本地推理优化、多 agent
- 原因：避免范围失控，遵循 PDF 推荐开发顺序

## D-003 存储边界使用 SQLite + Milvus Lite + 文件系统
- 日期：2026-03-25
- 决策：结构化数据进 SQLite，向量进 Milvus Lite，原始文件进 `data/uploads`
- 原因：职责清晰，便于逐步替换和调试

## D-004 文档主语言中文，代码与命名英文
- 日期：2026-03-25
- 决策：项目文档以中文为主，目录和代码标识使用英文
- 原因：兼顾沟通效率和工程可维护性

## D-005 初版使用规则驱动 intent 和本地哈希 embedding
- 日期：2026-03-25
- 决策：先不引入外部 LLM 或在线 embedding API
- 原因：确保最小系统可离线本地运行，先验证 graph、工具和存储骨架

## D-006 当前环境下向量层采用 Milvus 优先、文件索引回退
- 日期：2026-03-25
- 决策：保留 `MilvusLiteStore` 接口，但在当前 Python 3.13 环境中若 `milvus-lite` 不可安装，则自动回退到本地文件向量索引
- 原因：当前环境存在依赖分发限制，需要先保证骨架可运行，同时不改变后续回切 Milvus Lite 的接口边界

## D-007 在阶段 2 之前插入 VS Code 环境和版本日志基础设施
- 日期：2026-03-25
- 决策：先补齐 VS Code 运行配置、bootstrap 脚本和 `logs/` 版本更新记录机制，再进入阶段 2
- 原因：当前优先需求是本地 IDE 可直接运行，并且每轮迭代都留下可追踪的文本记录

## D-008 阶段 2 第一刀先做交互式 CLI 和 destructive 确认节点
- 日期：2026-03-26
- 决策：先实现 REPL + slash 命令，再把 `note delete` / `remind cancel` 接入 graph 级确认节点
- 原因：这是当前最小但高价值的阶段 2 切入点，能先提升真实使用体验，同时把状态图往“稳定、清晰、可扩展”推进一步

## D-009 会话记忆仅保留在当前终端进程
- 日期：2026-03-26
- 决策：交互式 CLI 的 `messages` 和 `pending_confirmation` 只保留在当前 REPL 进程，不做持久化恢复
- 原因：先把会话骨架和确认流跑通，避免过早扩展到历史恢复和会话存储

## D-010 retrieval filter 第一版采用自动 metadata + 本地过滤
- 日期：2026-03-26
- 决策：文件导入时自动记录 `file_id`、`source_name`、`extension`、`media_type`，检索时先做向量召回，再在本地按 metadata 过滤
- 原因：这是当前最稳妥的阶段 2 下一刀，能先把检索控制能力补齐，又不要求立刻重做向量存储结构
